# Настройка retention для экономии места в БД

Рекомендуемые **сокращённые** сроки хранения (меньше 90 дней) для минимизации объёма на Railway.

## Сроки по таблицам

| Таблица    | Срок    | Обоснование |
|------------|---------|-------------|
| **trades** | 30 дней | Фичи (rolling 60, RSI 14, MACD) и ML нуждаются в истории; 30 дней — безопасный минимум. 21 день возможен, но на малоактивных рынках качество может упасть. |
| **orderbook** | **7 дней** | В коде используется только для флага «есть ликвидность» (DISTINCT market_id). Длинная история снимков не нужна. |
| **features** | **14 дней** | Пересчитываются из trades каждый цикл; для API/дашборда достаточно недавних. |
| **news** | **7 дней** | Только лента и дедуп по link/title. |
| **signals** | 21 день | Бэктестер читает tail(5000); 21 день даёт достаточно точек. |
| **results** | **14 дней** | В API для графика PnL — LIMIT 500; 14 дней хватает. |

## От чего избавляемся в первую очередь

- **orderbook** — режем до 7 дней (максимальная экономия при минимальном влиянии).
- **news** — 7 дней.
- **results** — 14 дней.
- **features** — 14 дней.
- **signals** — 21 день.
- **trades** — 30 дней (ниже не рекомендуется без тестов на своих рынках).

## Реализация в конфиге

Вариант с двумя переменными:

- `DATA_RETENTION_DAYS=30` — trades, signals.
- `DATA_RETENTION_DAYS_SHORT=7` — orderbook, news.

И зашить в код константы для features (14) и results (14), либо ввести третью переменную `DATA_RETENTION_DAYS_MEDIUM=14`.

В `cleanup_retention()` вызывать удаление с соответствующим cutoff для каждой таблицы.
